Patrones de Diseño

Soluciones recurrentes a problemas comunes en el diseño de software

Tema 3.2 del programa Basado en presentación del curso Estudio activo

¿Qué es un patrón de diseño?

Los patrones de diseño son soluciones o técnicas típicas a problemas comunes en el diseño de software. Son como planos prediseñados que puedes personalizar para resolver problemas recurrentes en tu código.

Idea clave

El patrón no es un fragmento de código específico, sino un concepto general para resolver un problema específico. Cada patrón describe una estructura que se adapta al contexto.

Clasificación por propósito

Los patrones varían en complejidad, nivel de detalle y escala de aplicabilidad. Se clasifican en tres grandes familias:

Creational

Creacionales

Proporcionan mecanismos de creación de objetos que aumentan la flexibilidad y reutilización del código existente.

  • Cómo crear objetos eficientemente
  • Ocultar la lógica de instanciación
  • Reducir el acoplamiento

Vistos en clase: Singleton, Factory Method, Abstract Factory

Structural

Estructurales

Explican cómo ensamblar objetos y clases en estructuras más grandes, manteniéndolas flexibles y eficientes.

  • Ensamblar objetos en estructuras
  • Garantizar flexibilidad ante cambios
  • Si una parte cambia, no se rompe el todo

Vistos en clase: Adapter, Facade

Behavioral

Comportamiento

Se encargan de la comunicación efectiva y la asignación de responsabilidades entre objetos.

  • Comunicación entre objetos
  • Asignación de responsabilidades
  • Algoritmos intercambiables

Vistos en clase: Observer

Patrones Creacionales

Mecanismos de creación de objetos que aumentan flexibilidad y reutilización del código existente.

Singleton Creacional

Asegura que una clase tenga una única instancia, y proporciona un punto de acceso global a dicha instancia.

Implementación · pasos clave

  1. Añadir un atributo static privado a la clase para almacenar la instancia Singleton (de tipo Singleton).
  2. Declarar el constructor como private. El método estático seguirá pudiendo invocarlo.
  3. Declarar un método estático público (getInstance()) que: verifica si la instancia ya fue creada y la devuelve, o la crea si todavía no existe.

Sintaxis

public class GameConfig {
    // 1) atributo estático privado
    private static GameConfig instance;

    private String difficulty;
    private int lives;
    private int enemySpeed;

    // 2) constructor privado
    private GameConfig() { }

    // 3) punto de acceso global
    public static GameConfig getInstance() {
        if (instance == null) {
            instance = new GameConfig();
        }
        return instance;
    }

    // get / set / showConfig...
}

Problema

En un videojuego con configuraciones globales (dificultad, jugadores, velocidad de enemigos) cada subsistema podría crear su propia GameConfig, generando inconsistencias y reglas impredecibles.

Solución

El Singleton garantiza que GameConfig sea única y accesible globalmente: todos los módulos llaman a GameConfig.getInstance() y reciben la misma instancia. Las reglas no pueden divergir.

Casos de uso

Configuración global de una aplicación · Conexiones a bases de datos · Loggers · Caches compartidos

Aplicabilidad

Utiliza Singleton cuando una clase de tu programa solo debe tener una instancia disponible para todos los clientes; por ejemplo, un único objeto de base de datos compartido por distintas partes del programa.

Práctica guiada del curso

  1. Crear GameConfig (atributos difficulty, lives, enemySpeed, set/get, método showConfig()) implementando Singleton.
  2. Crear EnemySystem (usa getEnemySpeed) y PlayerSystem (usa getLives).
  3. Cada sistema obtiene la configuración con GameConfig.getInstance().
  4. En Main: configurar valores, obtener la configuración global, crear los dos sistemas y demostrar que ambos comparten la misma instancia.

Factory Method Creacional

Proporciona una interfaz para crear objetos en una superclase, pero permite que las subclases decidan qué tipo de objeto instanciar.

Participantes

Producto
Interfaz común para los objetos creados por el Factory Method.
ProductoConcreto
Implementa la interfaz Producto.
Creador
Clase abstracta que declara el método de fábrica abstracto (crearProducto()), que devuelve un Producto.
CreadorConcreto
Sobrescribe el método de fábrica para devolver un ProductoConcreto.

Sintaxis

// Producto
public interface Payment {
    void processPayment(double amount);
}

// Productos concretos
public class CreditCardPayment implements Payment {
    public void processPayment(double amount) {
        System.out.println("Pago con tarjeta: " + amount);
    }
}

public class PaypalPayment implements Payment {
    public void processPayment(double amount) {
        System.out.println("Pago con PayPal: " + amount);
    }
}

// Creador (abstracto)
public abstract class PaymentFactory {
    public abstract Payment createPayment();
}

// Creadores concretos
public class CreditCardFactory extends PaymentFactory {
    public Payment createPayment() {
        return new CreditCardPayment();
    }
}

public class PaypalFactory extends PaymentFactory {
    public Payment createPayment() {
        return new PaypalPayment();
    }
}

Problema

Una aplicación de gestión logística que solo soporta transporte en camión, con la mayor parte del código dentro de la clase Camión. Cuando llegan solicitudes para transporte marítimo, hay que refactorizar para permitir nuevos tipos sin alterar la estructura existente.

Solución

En lugar de llamar al operador new para construir objetos directamente, se invoca a un método fábrica especial. Las subclases pueden sobrescribirlo para devolver el tipo específico que necesitan.

Aplicabilidad

Utiliza Factory Method cuando no conozcas de antemano las dependencias y los tipos exactos de los objetos con los que deba funcionar tu código.

Práctica guiada del curso

  1. Sistema de procesamiento de pagos aplicando el patrón Factory.
  2. Métodos de pago soportados: Tarjeta de crédito y PayPal, con un processPayment().
  3. Crear los productos abstractos y concretos, y lo mismo para la fábrica.
  4. Main: probar ambas fábricas.

Abstract Factory Creacional

Crea familias de objetos relacionados que deben ser compatibles entre sí, sin especificar sus clases concretas.

Participantes

FábricaAbstracta
Interfaz para operaciones que crean objetos producto abstractos.
FábricaConcreta
Implementa las operaciones para crear objetos producto concretos.
ProductoAbstracto
Interfaz para un tipo de objeto producto.
ProductoConcreto
Define un objeto producto creado por la fábrica correspondiente.
Cliente
Solo usa interfaces de FábricaAbstracta y ProductoAbstracto.

Sintaxis

// Productos abstractos
public interface Moneda { void mostrar(); }
public interface BloqueInterrogante { void mostrar(); }

// Productos concretos — familia GameBoy
public class GameBoyMoneda implements Moneda {
    public void mostrar() {
        System.out.println("Moneda en GameBoy");
    }
}
public class GameBoyBloqueInterrogante implements BloqueInterrogante {
    public void mostrar() {
        System.out.println("Bloque ? en GameBoy");
    }
}

// Productos concretos — familia NintendoDS (análogo)

// Fábrica abstracta
public interface AbstractFactory {
    Moneda createMoneda();
    BloqueInterrogante createBloqueInterrogante();
}

// Fábricas concretas
public class GameBoyItemFactory implements AbstractFactory {
    public Moneda createMoneda() {
        return new GameBoyMoneda();
    }
    public BloqueInterrogante createBloqueInterrogante() {
        return new GameBoyBloqueInterrogante();
    }
}

Problema

Un simulador de tienda de muebles con (1) una familia de productos relacionados — Silla + Sofá + Mesa de centro — y (2) varias variantes — Moderna, Victoriana, Art Déco. Los clientes se enfadan cuando reciben muebles que no combinan.

Solución

Primero, crear interfaces para cada producto (Silla, Sofá, Mesilla). Luego declarar una Fábrica abstracta con métodos para crear todos los productos de la familia (crearSilla, crearSofá, crearMesilla). Para cada variante (Moderna, Victoriana...) crear una clase de fábrica independiente: la FábricaDeMueblesModernos solo crea objetos modernos.

Abstract Factory vs Factory Method

Abstract FactoryFactory Method
Define una interfaz para crear múltiples productos relacionados. Delega la creación de un solo producto a subclases.
Garantiza que los productos sean compatibles (todos premium o todos estándar). Decide qué clase concreta instanciar.

Práctica guiada del curso (Nintendo)

  1. Interfaz AbstractFactory con createMoneda() y createBloqueInterrogante().
  2. Productos concretos: GameBoyMoneda, GameBoyBloqueInterrogante, NintendoDSMoneda, NintendoDSBloqueInterrogante.
  3. Fábricas concretas: GameBoyItemFactory, NintendoDSItemFactory (cada una crea los productos de su familia).
  4. Main: probar ambas fábricas demostrando que el cliente trabaja con diferentes familias sin depender de clases concretas.

Ejercicio extra: fábrica de chocolates (Premium / Estándar) produciendo Tabletas, Bombones y Barras.

Patrones Estructurales

Cómo ensamblar objetos y clases en estructuras más grandes manteniéndolas flexibles y eficientes. Si una parte cambia, no es necesario cambiar toda la estructura.

Adapter Estructural

Convierte la interfaz de una clase en otra distinta que es la que esperan los clientes. Permite que cooperen clases que de otra manera no podrían por tener interfaces incompatibles, sin modificar la clase original.

Idea central

Hacer que dos clases que no hablan el mismo "idioma" puedan trabajar juntas.

Participantes

Cliente
La parte del programa que quiere usar un objeto.
Objetivo-Cliente (Interfaz)
Define la interfaz específica del dominio que usa o quiere el Cliente.
Adaptable
Clase existente que necesita ser adaptada. Es lo que ya se tiene y no se puede cambiar ("vieja").
Adaptador
Adapta la interfaz del Adaptable a la interfaz Objetivo.

Sintaxis

// Interfaz Objetivo: lo que el cliente espera
public interface Messenger {
    void sendMessage(String msg);
}

// Clases que ya cumplen el contrato
public class EmailSender implements Messenger {
    public void sendMessage(String msg) {
        System.out.println("Email: " + msg);
    }
}

// Clase externa INCOMPATIBLE — no la podemos modificar
public class WhatsAppAPI {
    public void sendWhatsApp(String phoneNumber, String text) {
        System.out.println("WhatsApp a " + phoneNumber + ": " + text);
    }
}

// El ADAPTADOR: implementa la interfaz objetivo y envuelve a la externa
public class WhatsAppAdapter implements Messenger {
    private WhatsAppAPI api;
    private String phoneNumber;

    public WhatsAppAdapter(String phoneNumber) {
        this.phoneNumber = phoneNumber;
        this.api = new WhatsAppAPI();
    }

    public void sendMessage(String msg) {
        api.sendWhatsApp(phoneNumber, msg);   // traduce la llamada
    }
}

Problema

Una aplicación de monitoreo del mercado de valores que descarga información de bolsa en formato XML. Al querer integrar una biblioteca de análisis, descubres que solo funciona con datos en JSON. La biblioteca es externa: no se puede modificar.

Solución

Crear un adaptador: un objeto especial que convierte la interfaz de un objeto, de forma que otro objeto pueda comprenderla.

Analogía del mundo real

Cuando viajas de Europa a Estados Unidos, los enchufes son diferentes y un enchufe español no sirve. El problema se resuelve con un adaptador que incluye el enchufe americano y el europeo.

Aplicabilidad

Utiliza la clase adaptadora cuando quieras usar una clase existente, pero su interfaz no sea compatible con el resto del código. También cuando quieras reutilizar varias subclases existentes que carezcan de alguna funcionalidad común que no pueda añadirse a la superclase.

Práctica guiada del curso (WhatsApp)

  • Messenger: interfaz común con sendMessage(String msg).
  • EmailSender y SMSSender: implementan Messenger.
  • WhatsAppAPI: externa, con sendWhatsApp(String phoneNumber, String text).
  • WhatsAppAdapter: adapta WhatsAppAPI a Messenger, recibe phoneNumber por constructor.
  • Main: referencias de tipo Messenger y llamadas a sendMessage sobre cada una.

Facade Estructural

Proporciona una interfaz unificada para un conjunto de interfaces de un sistema, escondiendo su complejidad.

Participantes

Facade
Sabe quiénes son las clases ante una petición y delega las peticiones a los objetos apropiados.
Subsistema
Implementa la funcionalidad del subsistema; realiza las labores encomendadas por el Facade.
Cliente
Habla solo con el Facade. No conoce los subsistemas.

Analogía del mundo real

Cuando llamas a una tienda para hacer un pedido, el operador es tu fachada a todos los servicios y departamentos. Te da una sencilla interfaz de voz al sistema de pedidos, pasarelas de pago y servicios de entrega.

Sintaxis

// Subsistemas (cada uno con su responsabilidad)
public class CPU {
    public void freeze()  { System.out.println("CPU: freeze"); }
    public void jump()    { System.out.println("CPU: jump"); }
    public void execute() { System.out.println("CPU: execute"); }
}
public class HardDrive {
    public void read() { System.out.println("HD: read"); }
}
public class Memory {
    public void load() { System.out.println("Memory: load"); }
}

// LA FACHADA: una interfaz simple que orquesta los subsistemas
public class ComputerFacade {
    private CPU cpu;
    private HardDrive hd;
    private Memory memory;

    public ComputerFacade() {
        this.cpu = new CPU();
        this.hd  = new HardDrive();
        this.memory = new Memory();
    }

    public void start() {
        cpu.freeze();
        memory.load();
        hd.read();
        cpu.jump();
        cpu.execute();
        System.out.println("La computadora inició");
    }
}

Práctica guiada del curso (Restaurante)

  • RestaurantFacade con método realizarPedido(String cliente, String platillo).
  • Subsistemas:
    • MesaService.verificarMesaDisponible()
    • CocinaService.registrarOrden(String platillo)
    • FacturaService.generarFactura(String cliente, String platillo)
    • NotificacionService.enviarNotificacion(String cliente)
  • El Main solo habla con la fachada, no con los subsistemas.

Ejercicio extra: ComputerFacade con CPU, HardDrive y Memory, método start() que orquesta todo.

Patrones de Comportamiento

Comunicación efectiva y asignación de responsabilidades entre objetos.

Observer Comportamiento

Define una dependencia uno-a-muchos entre objetos, de forma que cuando un objeto cambia de estado se notifica y actualiza automáticamente a todos los objetos que dependen de él.

Participantes

Observer (Subscriber)
Interfaz con el método para actualizar los objetos que deben ser notificados ante cambios del sujeto.
Subject (Publisher)
Interfaz con métodos para agregar, eliminar y notificar observadores cuando el estado cambie.
ObservadorConcreto
Mantiene una referencia al SujetoConcreto. Guarda un estado consistente con el del sujeto e implementa la interfaz de actualización.
SujetoConcreto
Almacena el estado de interés, implementa Sujeto, contiene la lista de observadores y un método que indica el cambio de estado.

Sintaxis

// Observer
public interface Observer {
    void update(String message);
}

// Subject (Publisher)
public interface Subject {
    void addObserver(Observer o);
    void removeObserver(Observer o);
    void notifyObservers();
}

// Sujeto concreto
public class YoutubeSubject implements Subject {
    private String state;
    private List<Observer> listObservers = new ArrayList<>();

    public void addObserver(Observer o) { listObservers.add(o); }
    public void removeObserver(Observer o) { listObservers.remove(o); }

    public void notifyObservers() {
        for (Observer o : listObservers) {
            o.update(state);
        }
    }

    public void newVideo(String state) {
        this.state = state;
        notifyObservers();
    }
}

// Observador concreto
public class YoutubeObserver implements Observer {
    public void update(String message) {
        System.out.println(message);
    }
}

Problema

Tienes dos tipos de objetos: Cliente y Tienda. El cliente está muy interesado en un producto (un nuevo iPhone) que llegará pronto. Podría visitar la tienda cada día para comprobar disponibilidad, pero la mayoría de estos viajes serían en vano.

Solución

Al Sujeto se le añade: (1) un campo matriz para almacenar una lista de referencias a objetos suscriptores, y (2) varios métodos públicos que permiten añadir y eliminar suscriptores de esa lista. Cuando el estado cambia, el Sujeto notifica a todos los suscriptores.

Práctica guiada del curso (Redes sociales)

  • En Main: crear los Subject de YouTube e Instagram.
  • Usar los métodos addObserver, removeObserver, newReel (Instagram) y newVideo (YouTube).
  • En el método update: usar un System.out.println que imprima el mensaje recibido como parámetro.
  • Observadores: YoutubeObserver e InstagramObserver, ambos implementan Observer.

Cheatsheet — repaso rápido

Lo esencial de cada patrón, condensado para el día previo al examen.

Creacionales

Singleton

  • Propósito: una única instancia + acceso global.
  • Receta: atributo static privado + constructor private + getInstance() público estático.
  • Check: if (instance == null) instance = new ...
  • Casos: config global, conexión DB, logger.

Factory Method

  • Propósito: delegar la creación de un objeto a subclases.
  • Roles: Producto (interfaz), ProductoConcreto, Creador (abstracto), CreadorConcreto.
  • Sustituye: el operador new por un método de fábrica.
  • Usar cuando: no conoces de antemano los tipos exactos a instanciar.

Abstract Factory

  • Propósito: crear familias de objetos relacionados compatibles entre sí.
  • Roles: FábricaAbstracta + FábricasConcretas + ProductosAbstractos + ProductosConcretos + Cliente.
  • Garantiza: coherencia (todos modernos, todos premium...).
  • vs Factory Method: Abstract Factory crea varios productos; Factory Method crea uno solo.

Estructurales

Adapter

  • Propósito: hacer que dos interfaces incompatibles trabajen juntas.
  • Roles: Cliente, Objetivo (interfaz), Adaptable (la "vieja"), Adaptador.
  • Regla clave: no se puede modificar la clase Adaptable.
  • Receta: el Adaptador implementa la interfaz Objetivo y envuelve al Adaptable.
  • Analogía: enchufe europeo ↔ enchufe americano.

Facade

  • Propósito: interfaz unificada y simple para un subsistema complejo.
  • Roles: Facade + Subsistemas + Cliente.
  • Receta: el Facade tiene composición de los subsistemas y orquesta sus llamadas en métodos simples.
  • Analogía: el operador de la tienda telefónica.
  • Cliente: solo habla con la fachada, no con los subsistemas.

Comportamiento

Observer

  • Propósito: uno-a-muchos. Cuando el sujeto cambia, todos los observadores se enteran.
  • Roles: Subject/Publisher + ConcreteSubject + Observer/Subscriber + ConcreteObserver.
  • Métodos del Subject: addObserver, removeObserver, notifyObservers.
  • Método del Observer: update(message).
  • Receta: el sujeto tiene List<Observer> y al cambiar de estado recorre la lista llamando a update.

Mnemónicos

  • Creacionales = "¿cómo creo objetos?"
  • Estructurales = "¿cómo conecto/agrupo objetos?"
  • Comportamiento = "¿cómo se comunican los objetos?"
  • Singleton: uno y solo uno.
  • Factory: alguien más hace el new.
  • Adapter: traductor entre dos interfaces.
  • Facade: recepcionista del subsistema.
  • Observer: suscripción y notificación.

Errores comunes que evitar

Singleton

  • Olvidar declarar el constructor como private (rompe el patrón).
  • Olvidar el static en el atributo instance.
  • No verificar if (instance == null) dentro de getInstance().

Factory Method

  • Que la fábrica devuelva una clase concreta en lugar de la interfaz Producto.
  • Hacer new directo en el cliente en vez de pasar por la fábrica.

Adapter

  • Modificar la clase "vieja" en vez de crear el adaptador (eso ya no es Adapter).
  • El adaptador implementa la interfaz objetivo, no la extiende.

Observer

  • Que el Subject conozca tipos concretos de Observers en vez de la interfaz.
  • Olvidar llamar a notifyObservers() después de cambiar el estado.
